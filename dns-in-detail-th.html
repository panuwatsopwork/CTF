Task 1 What is DNS?

DNS (Domain Name System) คือระบบที่แปลง "ชื่อโดเมนที่คนจำง่าย" (เช่น tryhackme.com) ให้กลายเป็น "IP Address ที่เครื่องเข้าใจ" (เช่น 104.26.10.229)

🏠 เปรียบเทียบให้เห็นภาพ

ทุกบ้านมี "เลขที่บ้าน" ที่ไม่ซ้ำกัน ไว้ส่งจดหมายให้ถึงปลายทาง — เครื่องคอมพิวเตอร์ก็เช่นกัน มี IP Address ของตัวเอง แต่การจำตัวเลข 4 ชุดมันยาก DNS เลยทำหน้าที่เหมือน "สมุดโทรศัพท์ของอินเทอร์เน็ต"

DNS แปลงชื่อโดเมน → IP Address เพื่อให้เครื่องเชื่อมต่อได้

🎯 ทำไม DNS สำคัญ?

  • ไม่ต้องจำตัวเลข IP ที่ซับซ้อน
  • เว็บไซต์เปลี่ยน IP ได้โดยที่ผู้ใช้ไม่ต้องเปลี่ยน URL
  • รองรับการกระจายโหลด (Load Balancing) ผ่านหลาย IP ในชื่อเดียว

🏷 TLD (Top-Level Domain)

คือส่วน "ขวาสุด" ของชื่อโดเมน เช่น .com, .org, .th — แบ่งเป็น 2 แบบหลัก:

ประเภท TLDความหมายตัวอย่าง
gTLD
(Generic)		บอกวัตถุประสงค์ของโดเมน.com (commercial), .org (organisation), .edu (education), .gov (government)
ccTLD
(Country Code)		บอกประเทศที่จดทะเบียน.ca (Canada), .co.uk (UK), .th (Thailand)

ปัจจุบันมี TLD ใหม่ ๆ เพิ่มเรื่อย ๆ เช่น .online, .club, .website, .biz รวมแล้ว มากกว่า 2,000 TLDs

🏠 Second-Level Domain

คือส่วนที่อยู่ "ซ้ายของ TLD" เช่นใน tryhackme.com ส่วน tryhackme คือ Second-Level Domain

📏 กฎการตั้งชื่อ

  • ยาวได้ สูงสุด 63 ตัวอักษร (ไม่นับ TLD)
  • ใช้ได้แค่ a-z, 0-9 และเครื่องหมายขีดกลาง (-)
  • ห้าม ขึ้นต้น/ลงท้ายด้วยขีดกลาง
  • ห้าม มีขีดกลางติดกัน 2 ตัว

🔖 Subdomain

คือส่วนที่อยู่ "ซ้ายของ Second-Level Domain" คั่นด้วยจุด เช่น admin.tryhackme.com ส่วน admin คือ Subdomain

  • กฎตั้งชื่อเหมือน Second-Level Domain (63 ตัวอักษร, a-z 0-9 และขีดกลาง)
  • ซ้อน Subdomain หลายชั้นได้ เช่น jupiter.servers.tryhackme.com
  • ความยาวรวมทั้งชื่อโดเมน ห้ามเกิน 253 ตัวอักษร
  • จำนวน Subdomain ไม่มีขีดจำกัด

Task 3 Record Types

DNS ไม่ได้แค่ผูกชื่อกับ IP — มี Record Type หลายแบบเพื่อจัดการข้อมูลต่าง ๆ ของโดเมน

Recordหน้าที่ตัวอย่าง
A Record ผูกชื่อโดเมนกับ IPv4 Address tryhackme.com → 104.26.10.229
AAAA Record ผูกชื่อโดเมนกับ IPv6 Address tryhackme.com → 2606:4700:20::681a:be5
CNAME Record ชี้ไปยัง "ชื่อโดเมนอื่น" (alias) store.tryhackme.com → shops.shopify.com
MX Record บอก Server ที่ใช้รับ-ส่ง Email ของโดเมน มี Priority Flag เพื่อระบุลำดับ Server หลัก/สำรอง tryhackme.com MX → alt1.aspmx.l.google.com
TXT Record เก็บข้อความได้อิสระ ใช้สำหรับ SPF, DMARC, ยืนยันความเป็นเจ้าของโดเมน ฯลฯ "v=spf1 ip4:192.0.2.0/24 ~all"

💡 ตัวอย่าง TXT Record การใช้งานจริง

  • _acme-challenge.example.com — ยืนยันตัวตนตอนขอ SSL Certificate
  • v=spf1 ... — บอกว่า Server ใดมีสิทธิ์ส่งอีเมลแทนโดเมน (กัน spam/spoof)
  • v=DMARC1; p=reject — นโยบายจัดการอีเมลที่ไม่ผ่าน SPF/DKIM
  • MS=ms12345678 — Microsoft ใช้ยืนยันความเป็นเจ้าของโดเมน

🎯 จุดที่ต้องจำ

  • A = IPv4 / AAAA = IPv6 (4 ตัว A เพราะ IPv6 ยาวกว่า)
  • CNAME = "นามแฝง" ชี้ไปชื่ออื่นก่อนค่อย resolve ต่อ
  • MX = Mail eXchange — สำหรับอีเมล
  • TXT = Text — ข้อความอะไรก็ได้

Task 4 Making A Request — เส้นทางของ DNS Query

เมื่อพิมพ์ชื่อโดเมนในเบราว์เซอร์ จะเกิดการเดินทางหา IP อย่างไร? มาดูทีละขั้น

1
Local Cache — เครื่องเช็คแคชของตัวเองก่อนว่าเคยถามชื่อนี้ไหม ถ้ามี → ตอบเลย
2
Recursive DNS Server — ถ้าไม่มี ส่งคำถามไปยัง DNS ของ ISP (หรือ 8.8.8.8 / 1.1.1.1) ซึ่งก็มีแคชของตัวเอง
3
Root DNS Server — ถ้า Recursive ไม่รู้คำตอบ ไปถาม Root Server ที่เป็น "กระดูกสันหลัง" ของ DNS
4
TLD Server — Root ชี้ไปยัง TLD Server (เช่น Server ของ .com) ที่รู้ว่า Authoritative Server ของโดเมนนั้นอยู่ที่ไหน
5
Authoritative Nameserver — เจ้าของข้อมูลตัวจริง ตอบ Record ที่ต้องการ (เช่น A Record)
6
ส่งกลับ + Cache — คำตอบส่งกลับมาตามทาง พร้อมเก็บแคชตาม TTL เพื่อใช้ครั้งถัดไป

เส้นทางเต็มของ DNS Resolution: Client → Recursive → Root → TLD → Authoritative → IP

⏱ TTL (Time To Live)

TTL คือเวลาเป็น วินาที ที่บอกว่าควร Cache คำตอบนานเท่าไหร่ก่อนต้องไปถามใหม่ — ช่วยลดภาระเครือข่ายและทำให้เว็บโหลดเร็วขึ้น

📚 ศัพท์สำคัญที่ต้องจำ

  • Recursive DNS Server = DNS ที่ ISP ให้มา หรือเลือกใช้เอง (เช่น Google 8.8.8.8, Cloudflare 1.1.1.1) ทำหน้าที่ "วิ่งตามหาคำตอบ" ให้
  • Root DNS Server = กระดูกสันหลังของอินเทอร์เน็ต ชี้ไปยัง TLD Server ที่ถูกต้อง
  • TLD Server = เก็บข้อมูลว่า Authoritative Server ของแต่ละโดเมนใต้ TLD อยู่ที่ไหน
  • Authoritative / Nameserver = "เจ้าของข้อมูลตัวจริง" ของโดเมน ที่ใช้แก้ไข DNS Record

💡 ตัวอย่าง: tryhackme.com

Nameserver ของ tryhackme.com คือ kip.ns.cloudflare.com และ uma.ns.cloudflare.com — มีหลายตัวเพื่อ Backup หากตัวหลักล่ม

Task 5 Practical — ทดลอง Query DNS

Task นี้บน TryHackMe มีหน้าเว็บให้ลองสร้าง DNS Query แบบต่าง ๆ พร้อมแสดงผลลัพธ์และคำสั่งที่ใช้รันบนเครื่องของตัวเอง

🧪 สิ่งที่ลองทำได้ใน Lab

  • Query CNAME ของ shop.website.thm
  • อ่านค่า TXT ของ website.thm
  • ดู Priority ของ MX Record
  • หา A Record (IPv4) ของ www.website.thm

🛠 คำสั่งที่ควรรู้จัก (บนเครื่องตัวเอง)

  • nslookup tryhackme.com — query แบบเร็ว ใช้ได้ทั้ง Windows/Linux/Mac
  • dig tryhackme.com — เครื่องมือมาตรฐานของ Linux/Mac แสดงข้อมูลละเอียดมาก
  • dig MX tryhackme.com — เจาะจง Record Type
  • dig @8.8.8.8 tryhackme.com — เลือก DNS Server ที่จะถาม

🎓 เป้าหมายของ Lab นี้

ฝึก "อ่านผลลัพธ์ DNS" จริง ๆ เพื่อให้คุ้นเคยกับโครงสร้าง Record และเป็นพื้นฐานก่อนใช้เครื่องมืออย่าง Wireshark, Burp Suite หรือทำ DNS Recon ใน Pentest

🧠 Key Takeaways

  • DNS = แปลงชื่อโดเมน → IP Address (สมุดโทรศัพท์ของอินเทอร์เน็ต)
  • โครงสร้างโดเมน: Subdomain . Second-Level Domain . TLD — อ่านจากขวาไปซ้าย
  • Subdomain ยาวได้ 63 ตัวอักษร / โดเมนรวมไม่เกิน 253 ตัวอักษร
  • Record หลัก: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), TXT (text)
  • การ Resolve: Client → Recursive → Root → TLD → Authoritative → Answer
  • TTL = เวลา Cache เป็นวินาที ลดภาระและทำให้เว็บโหลดเร็ว

🚀 นำไปต่อยอด

🔐 Cybersecurity

  • DNS Recon — รวบรวม Subdomain ของเป้าหมาย
  • วิเคราะห์ SPF / DMARC ใน TXT Record
  • ตรวจ DNS Hijacking / Cache Poisoning
  • ใช้ dig, dnsenum, amass ในการสำรวจ

👨‍💻 Web Developer / Sysadmin

  • ตั้งค่า A / CNAME ชี้โดเมนไป Server
  • ตั้ง MX Record สำหรับ Email Service
  • เพิ่ม TXT สำหรับ SPF / DKIM / DMARC
  • ปรับ TTL ก่อนย้าย Server เพื่อให้ Cache หมดเร็ว

📝 Checklist ก่อนไปบทเรียนถัดไป

  • อธิบายได้ว่า DNS คืออะไรและทำไมจึงสำคัญ
  • แยกได้ระหว่าง TLD, Second-Level Domain และ Subdomain
  • จำ Record Types หลักได้: A, AAAA, CNAME, MX, TXT
  • เล่าเส้นทางของ DNS Query ตั้งแต่ Client → Authoritative ได้
  • เข้าใจบทบาทของ TTL กับการ Caching
