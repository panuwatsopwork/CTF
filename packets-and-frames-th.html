Task 1 Packets และ Frames คืออะไร?

Packets และ Frames คือ "ชิ้นข้อมูลเล็ก ๆ" ที่เมื่อนำมาประกอบกันจะกลายเป็นข้อความหรือข้อมูลขนาดใหญ่ที่สมบูรณ์ แต่ทั้งสองคำนี้อยู่ใน Layer ที่ต่างกันใน OSI Model

คำศัพท์OSI Layerมีอะไรบ้าง
PacketLayer 3 (Network)IP Header + Payload (เน้นเรื่อง IP Address)
FrameLayer 2 (Data Link)ห่อ Packet อีกทีและเพิ่ม MAC Address

🏠 เปรียบเทียบกับการส่งจดหมาย

ลองนึกถึงการส่งจดหมาย — ซองจดหมาย คือ Frame ที่ใช้พาเนื้อหาเดินทางไปยังที่หมาย ส่วน กระดาษจดหมายข้างใน คือ Packet เมื่อผู้รับเปิดซอง (Frame) ออก ก็จะรู้ว่าต้องทำอะไรต่อกับเนื้อหา (Packet) ภายใน

กระบวนการห่อข้อมูลแบบนี้เรียกว่า Encapsulation

โครงสร้าง: Frame ห่อหุ้ม Packet โดยเพิ่ม MAC Header / Trailer

ทำไมต้องแบ่งข้อมูลเป็น Packets?

การส่งข้อมูลเป็นชิ้นเล็ก ๆ มีข้อดีหลายอย่าง โดยเฉพาะ ลดปัญหาคอขวด (bottleneck) ในเครือข่าย เพราะหากส่งข้อมูลก้อนใหญ่ทีเดียว เครือข่ายจะติดขัด

🖼️ Use Case: การโหลดรูปภาพจากเว็บไซต์

เมื่อคุณเปิดเว็บไซต์และมีรูปแมวอยู่ในหน้า รูปนั้นจะ ไม่ถูกส่งมาเป็นไฟล์ก้อนเดียว แต่จะถูกแบ่งเป็น Packet หลาย ๆ ชิ้น แล้วส่งทยอยมา เมื่อถึงเครื่องคุณก็จะถูกประกอบกลับเป็นรูปภาพที่สมบูรณ์

📋 Headers สำคัญใน IP Packet

Headerหน้าที่
Time to Live (TTL)กำหนดอายุของ Packet เพื่อไม่ให้วน loop ในเครือข่ายตลอดไปหากหาปลายทางไม่เจอ
Checksumใช้ตรวจสอบความถูกต้อง (integrity) ของข้อมูล ถ้าค่าผิดเพี้ยน = ข้อมูลเสีย
Source AddressIP Address ของผู้ส่ง (ใช้ตอบกลับ)
Destination AddressIP Address ของปลายทาง (รู้ว่าต้องไปไหน)

🎯 จุดสำคัญที่ต้องจำ

  • Packet = มี IP Address (Layer 3)
  • Frame = มี MAC Address (Layer 2) ห่อ Packet อีกที
  • การแบ่งข้อมูลเป็น Packet ช่วยลดความหนาแน่นในเครือข่าย

Task 2 TCP/IP และ Three-Way Handshake

TCP (Transmission Control Protocol) คือกฎการสื่อสารในเครือข่ายที่ได้รับความนิยมที่สุดตัวหนึ่ง โครงสร้างจะคล้ายกับ OSI Model แต่ TCP/IP จะย่อให้เหลือ 4 Layers

Layer ใน TCP/IPหน้าที่โดยย่อ
1. Applicationโปรแกรมที่ผู้ใช้สัมผัส เช่น Browser, Email
2. Transportจัดการเรื่อง TCP/UDP, Port
3. Internetจัดการเรื่อง IP Address, Routing
4. Network Interfaceการส่งสัญญาณบนสายสัญญาณ/Wi-Fi

🔑 คุณสมบัติเด่นของ TCP

TCP เป็น Connection-based Protocol หมายความว่าก่อนจะส่งข้อมูล ต้องสร้างการเชื่อมต่อ ระหว่าง Client และ Server ก่อน วิธีที่ใช้คือ Three-Way Handshake ซึ่งทำให้ TCP รับประกันว่าข้อมูลจะถึงปลายทางจริง ๆ

✅ ข้อดีของ TCP

  • รับประกัน Integrity ของข้อมูล
  • มีการ Synchronise สองเครื่องเพื่อไม่ให้ข้อมูลมาสลับลำดับกัน
  • ทำกระบวนการต่าง ๆ เพื่อความน่าเชื่อถือสูง

❌ ข้อเสียของ TCP

  • ต้องการการเชื่อมต่อที่เสถียร ถ้าหายไป 1 ชิ้น = ส่งใหม่ทั้งหมด
  • การเชื่อมต่อช้าจะทำให้เป็นคอขวด เพราะจอง connection ทั้งหมด
  • ช้ากว่า UDP เพราะเครื่องต้องทำงานมากกว่า

📦 Headers สำคัญของ TCP Packet

Headerหน้าที่
Source PortPort ของผู้ส่ง (สุ่มจาก 0-65535 ที่ยังไม่ถูกใช้)
Destination PortPort ของปลายทาง เช่น Web Server ที่ Port 80 (ไม่สุ่ม)
Source IP / Destination IPIP ของผู้ส่ง / ปลายทาง
Sequence Numberเลขเริ่มต้นของข้อมูลแต่ละชิ้น สุ่มขึ้นมาตอนเริ่ม connection
Acknowledgement Numberหลังจากชิ้นแรกได้ Sequence แล้ว ชิ้นถัดไปจะเป็น Sequence + 1
Checksumใช้ตรวจสอบ Integrity ของข้อมูลด้วยการคำนวณทางคณิตศาสตร์
Dataข้อมูลจริง เช่น byte ของไฟล์ที่ส่ง
Flagใช้กำหนดพฤติกรรมของ Packet เช่น SYN, ACK, FIN

🤝 The Three-Way Handshake

เป็นกระบวนการ "จับมือ 3 ครั้ง" เพื่อสร้างการเชื่อมต่อระหว่างสองเครื่อง โดยใช้ข้อความพิเศษดังนี้:

ขั้นFlagคำอธิบาย
1SYNClient ส่งคำขอเริ่มต้นและส่ง Sequence Number ของตัวเอง
2SYN/ACKServer ตอบรับและส่ง Sequence Number ของตัวเองกลับ
3ACKClient ยืนยันว่ารับ Sequence ของ Server แล้ว → พร้อมรับ-ส่งข้อมูล
—DATAส่งข้อมูลจริงหลัง connection พร้อม
—FINปิด connection อย่างสุภาพ
—RSTปิดทันที (กรณีฉุกเฉิน/มีปัญหา)
การจับมือ 3 ครั้ง (Three-Way Handshake) ระหว่าง Alice และ Bob

🧮 Sequence Number ทำงานยังไง?

ข้อมูลที่ส่งจะถูกใส่ เลขลำดับสุ่ม และจะเพิ่มทีละ 1 ในแต่ละ Packet ถัดไป ทั้งสองเครื่อง ต้องตกลงกันในเลขลำดับนี้เพื่อให้ข้อมูลถูกประกอบในลำดับที่ถูกต้อง

  1. SYN — Client: "นี่คือเลขลำดับเริ่มต้นของฉัน (ISN = 0)"
  2. SYN/ACK — Server: "ของฉันคือ 5000 และฉันรับทราบของคุณ (0) แล้ว"
  3. ACK — Client: "รับทราบเลขของคุณ (5000) นี่คือข้อมูลที่ ISN+1 = 1"

🔚 การปิด Connection ของ TCP

เนื่องจาก TCP จองทรัพยากรของระบบไว้ การปิด connection ทันทีที่เสร็จงานจึงเป็น Best Practice โดยใช้ Flag FIN

กระบวนการปิด Connection ของ TCP

💡 Use Case: เปิดเว็บไซต์ในเบราว์เซอร์

ทุกครั้งที่คุณพิมพ์ https://google.com ในเบราว์เซอร์ เครื่องของคุณจะทำ Three-Way Handshake กับ Server ของ Google ก่อน จากนั้นถึงจะเริ่มดาวน์โหลด HTML, CSS, JavaScript จนหน้าเว็บแสดงผล เพราะข้อมูลเว็บไซต์ ต้องครบและเรียงลำดับถูกต้อง เลยต้องใช้ TCP

Task 4 UDP/IP — โปรโตคอลทางเลือกที่เร็วกว่า

UDP (User Datagram Protocol) เป็นอีกโปรโตคอลสำหรับการสื่อสารระหว่างเครื่อง แต่แตกต่างจาก TCP ตรงที่ UDP เป็น Stateless Protocol

⚡ ใจความสำคัญ

UDP ไม่ต้องสร้าง connection ก่อนส่งข้อมูล ไม่มี Three-Way Handshake ไม่มี Synchronisation — แค่ส่งออกไปเลย!

เมื่อไหร่ควรใช้ UDP?

UDP เหมาะกับสถานการณ์ที่ ยอมให้ข้อมูลหายได้บ้าง หรือ การเชื่อมต่อไม่เสถียรก็ยังพอใช้งานได้ เช่น Video Streaming, Voice Call, เกมออนไลน์

✅ ข้อดีของ UDP

  • เร็วกว่า TCP มาก
  • ปล่อยให้ Application ตัดสินใจเรื่อง flow control เอง ยืดหยุ่นสำหรับนักพัฒนา
  • ไม่จองการเชื่อมต่อต่อเนื่อง ใช้ทรัพยากรน้อย

❌ ข้อเสียของ UDP

  • ไม่สนใจว่าข้อมูลถึงปลายทางหรือไม่
  • หาก connection ไม่เสถียร → ผู้ใช้จะรู้สึกถึงคุณภาพที่ลดลง
  • ไม่มีการตรวจสอบ Integrity แบบ TCP

📦 Headers ของ UDP Packet

UDP มี Header น้อยกว่า TCP แต่ก็ใช้ Header พื้นฐานร่วมกัน:

Headerหน้าที่

Source / Destination AddressIP ต้นทาง / ปลายทาง
Source PortPort ของผู้ส่ง (สุ่ม 0-65535)
Destination PortPort ของบริการปลายทาง (ไม่สุ่ม)

UDP ส่งข้อมูลโดยไม่ต้องสร้างการเชื่อมต่อล่วงหน้า

📺 Use Case: Video Call กับ การโอนไฟล์

Video Call (Zoom, Discord): ใช้ UDP เพราะถ้าภาพหายไป 1 เฟรม เราอยากเห็นเฟรมถัด ๆ ไปเรียลไทม์ มากกว่ารอเฟรมเก่าที่หาย

ดาวน์โหลดไฟล์ (FTP, HTTP): ใช้ TCP เพราะถ้าข้อมูลหายไป 1 byte ไฟล์อาจเปิดไม่ได้เลย — ต้องการันตีว่าครบทุก byte

Task 5 Ports 101 — Port และ Protocol สำคัญ

Port คือ "จุดที่ข้อมูลถูกแลกเปลี่ยน" ภายในเครื่องคอมพิวเตอร์ — เป็นเลขจำนวนเต็ม 0 ถึง 65,535

🚢 เปรียบเทียบกับท่าเรือ

ลองนึกภาพ ท่าเรือ (Harbour) ที่มี ช่องจอด (Port) หลายช่อง เรือแต่ละแบบจะต้องจอดที่ช่องที่เหมาะกับขนาดและประเภทของตัวเอง — เรือสำราญลำใหญ่จอดที่ช่องของเรือประมงไม่ได้

เครือข่ายก็เหมือนกัน: Port ใช้บังคับ "กฎ" ว่าโปรแกรมไหนจะคุยที่ช่องไหน — ถ้าไม่เข้ากัน ก็คุยกันไม่ได้

🌐 ทำไมต้องมี "Standard Port"?

ถ้าใครก็เปิด Port อะไรก็ได้ตามใจ — โปรแกรมต่างเครื่องจะคุยกันไม่รู้เรื่อง ดังนั้นจึงมี มาตรฐานสากล เช่น Web ต้องใช้ Port 80 ทำให้ Browser ของทุกค่ายแปลข้อมูลแบบเดียวกันได้ (Chrome, Firefox, Edge, ...)

Port ที่อยู่ในช่วง 0–1024 เรียกว่า Common Ports ซึ่งจะถูกจองไว้สำหรับโปรโตคอลมาตรฐาน

📋 Common Protocols & Ports ที่ต้องจำ

ProtocolPortหน้าที่
FTP (File Transfer Protocol)21โอนไฟล์แบบ Client-Server
SSH (Secure Shell)22Login เข้าระบบแบบ text-based อย่างปลอดภัย
HTTP (HyperText Transfer Protocol)80พลังงานของ WWW — โหลดเว็บเพจ
HTTPS (Secure HTTP)443เหมือน HTTP แต่ เข้ารหัส
SMB (Server Message Block)445แชร์ไฟล์และอุปกรณ์ เช่น เครื่องพิมพ์

Server หนึ่งเครื่องเปิดได้หลาย Port เพื่อให้บริการคนละแบบ

🛠 Port ไม่จำเป็นต้องเป็นค่าเริ่มต้นเสมอไป

โปรโตคอลพวกนี้คือ มาตรฐาน เท่านั้น — คุณสามารถรัน Service บน Port อื่นได้ เช่น รัน Web Server บน Port 8080 แทน 80

⚠️ ข้อควรระวัง

ถ้ารัน Web Server บน Port ไม่มาตรฐาน Browser จะ เดาว่าเป็น 80 ดังนั้นคุณต้องระบุ Port หลัง URL ด้วย เช่น http://example.com:8080

🎯 Key Takeaways — สิ่งที่ต้องจำให้ได้

  • Packet อยู่ Layer 3 (มี IP), Frame อยู่ Layer 2 (มี MAC) และห่อ Packet
  • การส่งข้อมูลเป็นชิ้นเล็ก ๆ ช่วยลด Bottleneck และเพิ่มประสิทธิภาพ
  • TCP = ช้าแต่ชัวร์ (Connection-based, การันตี Integrity ด้วย Three-Way Handshake)
  • UDP = เร็วแต่ไม่การันตี (Stateless, เหมาะกับ Realtime)
  • Three-Way Handshake: SYN → SYN/ACK → ACK และปิดด้วย FIN
  • Port มีเลข 0–65535 และในช่วง 0–1024 คือ Common Ports ที่จองไว้ตามมาตรฐาน

🗺️ แผนภาพรวมเชื่อมโยงทุกแนวคิด

โมเดล TCP/IP 4 Layer และความเชื่อมโยงของ Packet/Frame/Port

  • Wireshark: ใช้ดักจับ Packet จริง ๆ บนเครือข่าย เพื่อวิเคราะห์ Three-Way Handshake
  • Nmap: สแกน Port เพื่อค้นหา Service ที่เปิดอยู่บนเครื่องเป้าหมาย
  • Netcat: สร้างการเชื่อมต่อ TCP/UDP ด้วยมือ เพื่อทดสอบ Service

👨‍💻 สำหรับนักพัฒนา

  • เลือก Protocol ให้เหมาะกับงาน — Realtime ใช้ UDP, การโอนข้อมูลสำคัญใช้ TCP
  • เข้าใจ Port ทำให้ตั้งค่า Server หรือ Firewall ได้แม่นยำขึ้น
  • Debug Network ได้รวดเร็วเมื่อเข้าใจกระบวนการ Handshake

📝 Checklist ก่อนไป Task ต่อไป

  • ☑ อธิบายความแตกต่างของ Packet กับ Frame ได้
  • ☑ บอกขั้นตอนการทำ Three-Way Handshake ได้
  • ☑ เปรียบเทียบ TCP กับ UDP ได้
  • ☑ จำ Port มาตรฐาน 6 ตัวได้: 21, 22, 80, 443, 445, 3389
  • ☑ เข้าใจว่าทำไมต้องมี Standardisation ใน Networking
